1.互联网协议

发表于 2018-09-04 | 分类于 Web

互联网的工作原理

互联网价值就在于信息互通，便于各方学术交流、思想碰撞。但既然要交流，就得像都能理解的同一种约定的信息媒介（语言、文字等），那么在互联网中同样的也需要双方都能理解的媒介，这就是互联网协议————“不以规矩，无以成方圆” 。

互联网传输协议分层——如何传输数据？

万丈高楼平地起，互联网协议并不是一开始就像现如今般完善。例如我们最熟悉的 HTTP 协议就是一个比较复杂的协议（应用层协议），也是经过一层层封装，才能设计得如此完备。

为了方便理解，将互联网的传输过程由下至上、由简入繁，一共分为五层：

  • 物理层 （最靠近硬件 0 or 1）
  • 链接层 （协议数据包结构：head + data）
  • 网络层 IP协议
  • 传输层 TCP协议
  • 应用层 （最靠近用户 自然语言）

互联网本质就是数据的传输，一切皆数据。那么数据如何传输呢？

互联网传输为分层传输 每一层有每一层需要遵守的协议。该分层模型设计得最棒的地方在于层与层之间解耦，上层的变动完全不涉及下层的结构。这是优秀框架的设计原则。

物理层 Physical Layer

  • 作用：该层就是负责把计算机连接起来的物理手段，作用是只负责传输 0 和 1 有序序列的电信号。（脑补了谍战片情报人员在电报机敲下的摩斯码）
  • 媒介：光纤、无线电波、电缆等

至于一连串 0 和 1 代表什么，who care？交给链路层解决。

链接层 Linked Layer

  • 引语：单纯的 0 和 1 没有任何实际意义，如果不创造出一个共同遵守的协议的话。所以接下来就必须规定解读方式：给庞大的电信号分组，多少个电信号算一组？组内每个信号位有何意义？
  • 作用：给电信号分组，
  • 方案：以太网协议（该层遵循的协议）：
    • 出现：每家公司都有自己的电信号分组方式。逐渐地，一种叫做”以太网”的协议，占据了主导地位，一统江湖。（目前局域网大多数都是使用以太网标准）
    • 以太网规定：一组电信号构成一个数据包，叫做”帧”（Frame）。每一帧分成两个部分：标头（Head）和数据（Data）。
      • Head：包含数据包的一些说明项，比如发送者、接收者、数据类型等。（我的理解这就是元数据：即为描述数据（Data）的数据。）
      • Data：真正的数据内容本身。
    • 广播：以太网协议规定，必须知道双方的MAC（Media Access Control）地址，才能将数据广播出去。对方对比标头的MAC地址，符合的才能够接收，否则丢弃数据包。
    • 局限：不可能让世界上每一个计算机都来接受这个数据包并判断。互联网是由很多个子网络组成的，你发出的数据包只能在所属的子网络进行广播。（那么如何提高效率？）
    • 问题：如果两台计算机所处的不是同一个子网络，则通过“路由”的方式，将数据路由出去。（类似于：同城有邮递员对着门牌号，不同城，只能放入邮递箱，给个目标地址的邮编），而子网络的地址就是邮编。
    • 解决方案：”网络层”的诞生。它的作用是引进一套新的地址，使得我们能够区分不同的计算机是否属于同一个子网络。这套地址就叫做”网络地址”，简称”网址”。 于是，”网络层”出现以后，每台计算机有了两种地址，一种是MAC地址，另一种是网络地址。两种地址之间没有任何联系，MAC地址是绑定在网卡上的，网络地址则是管理员分配的，它们只是随机组合在一起。
  • 解决的问题：有了传输内容（Head+Data）、传输地址（MAC地址）、传输方式（广播），”链接层”就可以在多台计算机之间传送数据了。
  • 实际的困难：理论上，单单依靠MAC地址，深圳的网卡就可以找到洛杉矶的网卡了，技术上是可以实现的。但是由于广播这种方式，会导致传输的效率极其低下，需要一个个去比对；而且除了接收者外的每台计算机都会接收到广播，然后丢弃数据包，这是灾难。有什么方案能解决这个问题？需要建立一个新的制度来控制数据包的大致走向，这就交给“网络层”来完成吧。

网络层 IP协议

  • 引导语：互联网是由无数的子网络共同组成的一个巨型网络，如果处在同一个子网络中（局域网 LAN）可以广播通信，如果处在不同的子网络中（广域网 WAN），怎么办？不可能“广播”啊，就要换一种方式——“路由”。“路由”：就是为了解决向不同的子网络分发数据包。
  • 方案：两台计算机互通信息，从逻辑上可以推断，协议头中包含两套地址，那么必定是先处理网络地址，然后再处理MAC地址。通常我们发送一个 http 请求，只知道对方的服务器地址，也就是知道对方的IP地址，但是我们是不知道对方服务器的MAC地址的，这就需要一个“IP to MAC”过程（ARP协议）
  • IP协议
    • 规定网络地址的协议叫做IP协议，定义的地址为IP地址。（有时访问一个服务器接口，服务器会去做IP地址过滤，不是白名单的不让访问，为了数据库的安全）
    • 版本：IPv4 IPv6
    • 本质：区分以太网协议中哪些计算机的MAC地址在同一个子网络中

IPv4

网络地址由32个二进制位组成，即4字节。

习惯上不用二进制标识，而是将每个字节用十进制表示IP地址

0.0.0.0-255.255.255.255

IP地址分成两个部分，前一部分代表网络(net)，后一部分代表主机(host)。比如，IP地址172.16.254.1，假定它的网络部分是 172.16.254，那么主机部分就是1。（一台主机标识唯一一台计算机，就可通过ARP协议解析出MAC地址）

处于同一个子网络的计算机，网络部分必定是相同的。

但是，问题在于如何划分IP地址的网络部分？这就要用到另一个参数”子网掩码”。

子网掩码：表示子网络特征的一个参数。

它在形式上等同于IP地址，也是一个32位二进制数字，它的网络部分全部为1，主机部分全部为0。比如，IP地址172.16.254.1，如果已知网络部分是前24位，主机部分是后8位，那么子网络掩码就是11111111.11111111.11111111.00000000，写成十进制就是255.255.255.0。

IP协议地址作用：

  • 每一台计算机分配地址
  • 确定不同IP地址是否在同一个子网络

根据IP协议发送的数据包就是IP数据包
本质：再一次封装，该层数据标头中，肯定有IP地址

IP地址有两套（IPv4 IPv6），IPv4 不够用，所以出现了IPv6

ARP协议 ———— 通过IP地址找到MAC地址

IP数据包是放在以太网数据包里发送的，所以我们必须同时知道两个地址。但是通常对方的IP地址是已知的（后文会解释），但是我们不知道它的MAC地址。所以，我们需要能够从IP地址得到MAC地址。

第一种情况，不同子网络；那么事实上没有办法得到对方的MAC地址，只能把数据包传送到两个子网络连接处的”网关”，让网关去处理，网关与网关之间的联系就是路由。

第二种情况，同一个子网络；那么我们可以用ARP协议，得到对方的MAC地址。
ARP协议也是发出一个数据包，其中包含它所要查询主机的IP地址，在对方的MAC地址这一栏，填的是FF:FF:FF:FF:FF:FF，表示这是一个”广播”地址。它所在子网络的每一台主机，都会收到这个数据包，从中取出IP地址，与自身的IP地址进行比较，两者相同，都做出回复，向对方报告自己携带的MAC地址；否则就丢弃这个包。

传输层 TCP协议

  • 导语：在网络层分析，我们已经可以在互联网上任意两台主机上建立通信。目前只是 Find Host √
  • 问题：接同一台主机上有许多程序（进程）都需要用到网络，比如，你一边浏览网页（浏览器进程），一边与朋友在线聊天（QQ进程）。当一个数据包发来的时候，你怎么知道，它是发送到网页的数据，还是发送到QQ的数据？所以我们还需要一个参数，表示这个数据包到底供哪个程序（进程）使用。
  • 主角：接下来要登场的就是期待了很久的“端口”，英文名“port”。
    端口：每一个使用网卡的程序进程编号。每个数据包都发到主机的特定端口，所以不同的程序就能取到自己所需要的数据。
    “端口”是0到65535之间的一个整数，正好16个二进制位。0到1023的端口被系统占用，用户只能选用大于1023的端口。
  • 传输层作用：就是建立”端口到端口”的通信。相比之下，”网络层”的功能是建立”主机到主机”的通信。只要确定主机和端口，我们就能实现程序之间的交流。因此，Unix系统就把主机+端口，叫做”套接字”（socket）。有了它，就可以进行网络应用程序开发了。

套接字socket = 主机host + 端口port

传输层有两套不同对立的协议：

UDP协议

简单点，通信双方一旦数据包发出，都无法知道对方是否收到。

TCP协议

简单点，就是为了解决 UDP 协议的问题。

这个协议非常复杂，但可以近似认为，它就是有确认机制的UDP协议，每发出一个数据包都要求确认。如果有一个数据包遗失，就收不到确认，发出方就知道有必要重发这个数据包了。

三次握手机制

应用层

应用程序收到”传输层”的数据，接下来就要进行解读。由于互联网是开放架构，数据来源五花八门，必须事先规定好格式，否则根本无法解读。

“应用层”的作用，就是规定应用程序的数据格式。

举例来说，TCP协议可以为各种各样的程序传递数据，比如Email、WWW、FTP等等。那么，必须有不同协议规定电子邮件、网页、FTP数据的格式，这些应用程序协议就构成了”应用层”。

这是最高的一层，直接面对用户。它的数据就放在TCP数据包的”数据”部分。
